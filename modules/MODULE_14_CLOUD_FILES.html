Module 14 — Cloud Files Internals

First Principle: The kernel is the final frontier — what you can't see can still kill you.

Overview

Cloud files (OneDrive, Dropbox, etc.) use a filter driver (CldFlt.sys) to intercept file operations and redirect them to cloud storage. This creates a new attack surface: reparse points, placeholder files, and sync root APIs that bridge userland and kernel.

Understanding this stack is critical for both offense (finding vulnerabilities) and defense (monitoring cloud file activity).

The Stack

Layer 0: NTFS

The underlying filesystem. Reparse points are NTFS metadata tags that tell the filter driver to intercept operations on this file.

Layer 1: CldFlt.sys (Filter Driver)

Kernel-mode driver that intercepts IRPs (I/O Request Packets) for files in registered sync roots. Handles hydration, dehydration, and placeholder management.

// Filter drivers attach to the filesystem stack
// They see EVERY operation before NTFS does
// Bug here = kernel privilege escalation

Layer 2: cldapi.dll (User API)

User-mode API for applications to interact with cloud files. Functions like CfCreatePlaceholders, CfUpdatePlaceholder, CfExecute.

Key insight: All cldapi.dll functions require a properly registered sync root. Random temp directories fail with ERROR_INVALID_PARAMETER.

Layer 3: OneDrive.exe (Sync Engine)

The user-mode application that manages sync, conflict resolution, and cloud communication.

Reparse Points

Reparse points are NTFS metadata that redirect file operations to a filter driver. When you see a cloud-stored file locally, it's actually a reparse point — the content lives in the cloud.

// Check if a file is a reparse point:
DWORD attributes = GetFileAttributesW(path);
if (attributes & FILE_ATTRIBUTE_REPARSE_POINT) {
    // This is a cloud file or junction
}

TOCTOU Race Conditions

Time-of-check vs time-of-use is especially dangerous in kernel drivers. If the file state changes between verification and operation, the driver may operate on stale data.

// Vulnerable pattern:
// 1. Check: Is file hydrated? (yes)
// 2. Race: Another thread dehydrates it
// 3. Use: Read file content (FAIL — file is gone)

// In kernel code, this can lead to:
// - Null pointer dereference
// - Use-after-free
// - Buffer overflow (wrong size assumed)

Driver Verifier

Driver Verifier is a Windows tool that stresses kernel drivers to catch bugs that would otherwise go silent. When enabled, it monitors pool allocations, IRQL levels, and API compliance.

CRITICAL: Driver Verifier can cause BSODs. That's the point — it turns silent corruption into immediate, debuggable crashes. Always test on a VM, never on production. 
// Enable Driver Verifier on CldFlt.sys:
verifier /standard /driver CldFlt.sys

// Reboot required
// After reboot, any bug in CldFlt.sys = BSOD
// Analyze with WinDbg:
!analyze -v
!verifier 3

Lab Exercise

Objective: Explore cloud files internals and find a reproducible anomaly.

  1. Enumerate sync roots on your machine: Get-ChildItem -Path $env:OneDrive -Recurse | Where-Object { $_.Attributes -match 'ReparsePoint' }
  2. Inspect a reparse point: fsutil reparsepoint query <file_path>
  3. Register a test sync root (requires admin): CfRegisterSyncRoot with proper CF_SYNC_ROOT_REGISTRATION
  4. Create a placeholder and observe: CfCreatePlaceholders with CF_PLACEHOLDER_CREATE_INFO
  5. Trigger a race condition: Thread A checks placeholder state, Thread B updates it, Thread A executes transfer
  6. Enable Driver Verifier on CldFlt.sys, reboot, and test
  7. Analyze any crashes with WinDbg: !analyze -v, !verifier 3
  8. BONUS: Find a reproducible crash and document the primitive
Test Environment: .42 has Driver Verifier active on CldFlt.sys. Any kernel bug will BSOD immediately. Document the crash and analyze with WinDbg. This is how real kernel research is done.
