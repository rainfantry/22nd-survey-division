$ # Cloud files architecture: # CldFlt.sys = filter driver (kernel) # cldapi.dll = user-mode API # OneDrive.exe = sync engine (user) # Sync root = registered directory tree
Layer 0: NTFS (underlying filesystem) Layer 1: CldFlt.sys (filter driver) Layer 2: cldapi.dll (user API) Layer 3: OneDrive.exe (sync engine) Reparse points mark cloud-stored files. Local files: no reparse tag. Cloud files: reparse tag + placeholder.
Understanding the stack is critical — fuzzing the wrong layer gives false positives.
$ # CfAPI fuzzing targets: # CfCreatePlaceholders — needs registered sync root # CfUpdatePlaceholder — updates file metadata # CfExecute — transfers data between local and cloud # CfHydratePlaceholder — downloads cloud file to local # CfDehydratePlaceholder — removes local copy, keeps cloud
API Fuzzing Results: CfCreatePlaceholders: Without sync root: 0x80070157 (ERROR_INVALID_PARAMETER) With sync root: SUCCESS (placeholder created) CfUpdatePlaceholder: Random paths: 0x80070057 (ERROR_INVALID_PARAMETER) Valid placeholder: SUCCESS CfExecute: Without proper handle: 0x80070057 With valid handle: SUCCESS CfHydratePlaceholder: Without placeholder: 0x80070006 (ERROR_INVALID_HANDLE) With placeholder: SUCCESS (file downloaded)
All APIs require proper sync root registration. Random temp directories fail.
$ # TOCTOU race condition theory: # Time-of-check: Verify file state (placeholder, hydrated, etc.) # Time-of-use: Perform operation on that file # Race window: Between check and use, file state changes
Race Condition Scenario: Thread A: Check if file is hydrated (yes) Thread B: Dehydrate the file (now it's not) Thread A: Read file content (FAIL — file is gone) In cloud files: Thread A: Check placeholder state Thread B: Update placeholder via CfUpdatePlaceholder Thread A: Execute transfer via CfExecute Result: Operation on stale state = undefined behavior
TOCTOU in kernel drivers is especially dangerous — can lead to memory corruption or privilege escalation.
